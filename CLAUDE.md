## Project

Unity 3D project — Editor version **6000.4.1f1 (Unity 6)** using the **Universal Render Pipeline (URP) 17.4.0**. Currently a small prototype with a third-person character controller and orbit camera in a single scene (`Assets/Scenes/SampleScene.unity`).

## Build / Run / Test

There is no command-line build script — this is a standard Unity project. Open the folder in the Unity Hub with editor `6000.4.1f1`. Builds and play-mode runs happen from the Editor.

The two C# projects (`Assembly-CSharp.csproj`, `Assembly-CSharp-Editor.csproj`) and `GameDev.sln` are **auto-generated by Unity** — never hand-edit them; they get rewritten on package refresh. Edit `.cs` files under `Assets/` and let Unity regenerate.

Headless build / test from the command line (if needed):

```powershell
& "C:\Program Files\Unity\Hub\Editor\6000.4.1f1\Editor\Unity.exe" -batchmode -quit -projectPath "H:\Gamedev\GameDev" -runTests -testPlatform EditMode -testResults "H:\Gamedev\GameDev\Logs\test-results.xml"
```

(Adjust the editor path to wherever the 6000.4.1f1 install lives on this machine.) The `com.unity.test-framework` package is installed, so EditMode/PlayMode tests are supported, but **no test assemblies exist yet** in the repo.

## Architecture

The gameplay code is two `MonoBehaviour` scripts that work as a pair, both attached to objects in `SampleScene`:

- **`Assets/Script/PlayerController.cs`** — Rigidbody character. **Camera-relative** WASD: caches `Camera.main.transform` in `Start`, flattens its forward/right to the XZ plane each `FixedUpdate`, and projects input onto that basis before assigning `rb.velocity` (Y preserved so gravity/jump work). Face-toward-move slerp. `Jump()` extracted to its own method — gated by `isGrounded` in `Update`, zeroes Y velocity before the impulse so jump height stays consistent regardless of fall speed. Ground check is `Physics.CheckSphere` at `transform.position + groundCheckOffset` against `groundLayer`, ignoring triggers, visualized via `OnDrawGizmosSelected` (green when grounded in play mode, red otherwise). Locks/hides the cursor in `Start`.
- **`Assets/Script/ThirdPersonCamer.cs`** — Class is `ThirdPersonCamera` (the **filename typo is intentional** — renaming invalidates the `.meta` GUID and breaks the scene's MonoBehaviour binding). Mouse-orbit follow camera: in `LateUpdate`, accumulates `Mouse X` → `yaw` and `-Mouse Y` → `pitch` (clamped), positions self at `target + Quaternion.Euler(pitch, yaw, 0) * (0, height, -distance)`, then `LookAt(target.position + Vector3.up * lookHeight)`. The camera-relative movement in `PlayerController` depends on this — if you replace the camera with a fixed-offset variant, also strip the `camForward`/`camRight` projection from `FixedUpdate` or controls will feel wrong.

The two scripts are coupled implicitly: `PlayerController` grabs `Camera.main.transform` at `Start` for its movement basis, and `ThirdPersonCamera.target` must be wired (in the scene/Inspector) to the player's transform. Neither serializes a reference to the other in code — the link lives in the scene file.

## Rendering

URP is configured with three quality tiers under `Assets/Settings/` — `URP-Performant`, `URP-Balanced`, `URP-HighFidelity` (each with a matching `*-Renderer.asset`). The active pipeline asset per quality level is wired in `ProjectSettings/QualitySettings.asset`. Global URP settings live in `Assets/UniversalRenderPipelineGlobalSettings.asset`. The volume profile for the sample scene is `Assets/Settings/SampleSceneProfile.asset`.

## Unity-specific gotchas for edits

- **Never delete or rename a `.meta` file** without doing the matching rename to its companion asset — Unity tracks asset identity by the GUID inside the `.meta`, and scene/prefab references break silently if it changes.
- Treat `Library/`, `Temp/`, `Logs/`, `obj/`, and the `*.csproj` / `*.sln` files as build artifacts. They regenerate; don't commit fixes into them.
- `Assets/TutorialInfo/` is leftover from the URP 3D template (the `Readme` ScriptableObject shown in a fresh project). Safe to ignore or delete; it isn't referenced by gameplay code.
